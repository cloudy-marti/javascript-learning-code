Exercice 1 : Document de base

  1. Que manque t'il comme information à la balise head pour que le fichier s'affiche correctement ?

    Il faut rajouter la balise <meta charset="utf-8"> pour que l'information s'affiche correctement.

  2. A quoi servent les tags h1, h2 et h3 ?

    Ces tags servent à marquer des titres (headers). Plus le nombre est petit, plus le titre sera gros ; cependant ceci peut etre changé ensuite via le fichier de style .css .

  3. A quoi sert le tag p ?

    Le tag p sert à envelopper les paragraphes "de base". Les tags HTML commencent par une balise ouvrante <foo> et finissent par une balise fermante </foo>.

  4. Comment mettre en gras ou en italique ?

    Il faut rajouter les balises <b> gras </b> et <i> italique </i>.

  5. Et comment faire pour mettre un mot à la fois en gras et en italique ?

    Il faut mettre le mot dans les deux balises ; la première balise ouvrante étant la dernière fermante : <b><i> gras et italique </i></b>.

  6. On souhaite que le titre de la fenêtre du navigateur soit "Compte Rendu de TP1", comment faire ?

    On rajoute le titre dans le tag <title> dans les balises <head>.

  7. Ajoutez un lien (le tag A) vers le cours avec comme titre cours de programmation web.

    Cours de Programmation Web

  8. Prenez une feuille de papier puis dessinez l'arbre des différents tags correspondant à votre compte rendu.

    L'arbre ci-dessous.

Exercice 2 : Uniform Resource Locator

  1. Ouvrez un navigateur Web et allez à l'URL suivante: ici

    • Quel est le nom de la machine à laquelle vous accédez ?

      www.w3.org

    • Quel est le nom de domaine auquel vous accédez ?

      org

  2. Si on exécute host w3.org, à quoi correspond la première ligne de la réponse ? À quoi correspond la troisième ligne de la réponse ?

    La première ligne de la réponse donne l'adresse IP de la machine à laquelle on veut accéder (dans ce cas 128.30.52.37). La troisième ligne de la réponse correspond au serveur mail de préference qui va recevoir les mails en SMTP envoyé vers la machine.

  3. Dans le navigateur Web, dans la barre contenant l'URL, remplacez le nom de la machine par son adresse IP pour vérifier que cela fonctionne toujours. A votre avis, pourquoi utilise-t-on des noms pour les machines plutôt que des adresses IP ?

    Remplacer l'url par son adresse IP va permettre au navigateur d'accéder au meme site (cependant, il faut l'accepter en exceptions car il ne vérifie pas les conditions de sécurité). On donne plutot le nom des machines parce que c'est plus simple pour l'etre humain de s'en rappeler.

  4. Exécutez la commande traceroute sur cette adresse IP.

    • Qu'affiche cette commande ?

      Elle affiche toutes les machines par lesquelles passe notre requete pour arriver jusqu'à sa destination.

    • Quel est le nom du routeur de sortie situé à l'université ?

      gwetud.u-pem.fr.

    • Comment s'appelle le réseau sur lequel est connecté l'université ?

      renater.fr.

    • Quels sont les autres réseaux qui sont traversés ?

      geant.net, zayo.com, akamai.com, mit.edu.

  5. On cherche maintenant à faire la même chose avec la machine hébergeant le cours http://forax.gitdub.io/course/progweb/. Que se passe-t-il si on essaye d'accéder à la page du cours dans le browser en remplaçant le nom de la machine par l'adresse IP trouvée précédemment ?

    On arrive jusqu'au site de gitdub mais le cours n'est pas affiché. (erreur 404), car gitdub héberge plusieurs sites et si on ne précise pas l'adresse, la machine ne sait pas quel site il faut afficher.

  6. On cherche à savoir si le cours est hébergé physiquement aux États-Unis, sachant qu'il est hébergé par la startup Californienne Gitdub.

    Cette machine est hébergée à New York (NY) aux Etats Units (US). Lorsqu'on fait un ping à imnh.org on obtient une moyenne de 80ms de latence, alors que lorsqu'on fait un ping au site du cours on obtient une moyenne de 9ms de latence. Ceci nous indique que la machine hébergeant le site ne se trouve pas au meme endroit que la machine qui héberge Gitdub.

Exercice 3 : Perroquet, quelle heure il est ?

  1. On cherche à savoir à quoi correspond le fichier /etc/services. Pour cela, dans le terminal tapez la commande man services

    Le fichier /etc/services fourni une correspondance entre les noms des services et leur numéro de port associés, ainsi que leurs protocoles. On a par exemple : socks - 1080/tcp

  2. Rappelez ce que fait la commande grep. Utilisez la commande grep pour déterminer le numéro du port TCP correspondant au service echo.

    La commande grep sert à chercher un mot dans un fichier ; si ce mot existe, elle va afficher sur la ligne standard la ligne où il se trouve.
    >> cat /etc/services | grep echo
    Le port TCP de la commande echo est le 7.

  3. Utilisez la commande nc pour tester le service echo sur la machine gaspard.univ-mlv.fr.

    Lorsqu'on écrit quelque chose sur le terminal, celui-ci affiche le message une seconde fois, en se comportant comme la commande echo.

  4. Faites la même manipulation avec le service daytime toujours sur la machine gaspard.univ-mlv.fr. Comment interpréter la réponse de la machine ?

    Le port TCP de la commande daytime est 13. Lorsqu'on fait un netcat sur la machine gaspard.univ-mlv.fr sur le port 13, on obtient l'heure et la date de la machine hébergeant ce site.

  5. Enfin, quel est le port TCP correspondant au service HTTP ?

    Port 80.

Exercice 4 : Service HTTP

  1. Quel est le header (l'entête, en français) de la réponse ?

    HTTP/1.1 200 OK
    Date: Thu, 29 Nov 2018 08:40:24
    Server: Apache
    Content-Length: 2957
    Connection: close
    Content-Type: text/html; charset=UTF-8

  2. A quoi correspondent les 3 valeurs sur la première ligne de l'entête ?

    • 1.1 est la version utilisée.
    • 200 : Tout c'est bien passé.

  3. Quel est le nom du serveur Web ?

    Apache

  4. A quoi correspond la valeur de Content-Type ?

    Cette valeur indique le type de fichier qui est lu en console et l'encodage UTF-8.

  5. A quoi correspond la valeur de Content-Length ?

    Il s'agit de la taille du fichier en octets.

  6. Faites la même analyse mais avec la requête GET /ens/toto.txt HTTP/0.9

    On a l'erreur 404 (le fichier n'existe pas) et la version est 1.1. Le serveur est apache et le content-type indique que le type de fichier espéré (un fichier de texte html). Comme le fichier n'existe pas, il n'y a pas de taille indiquée.

  7. Enfin, faites la même analyse avec la requête GET /index.html HTTP/0.9

    Le code de retour est 302 ; la page a été trouvée, mais elle se trouve ailleurs, la requete propose donc une redirection. La version est 1.1 et le serveur est Apache. La taille est de 279 octets et il s'agit d'un fichier texte de type html avec l'encodage iso-8859-1.

  8. À quoi correspond la valeur de Location ?

    Au site auquel on a été redirigés.
